# ContextWeave

ContextWeave is a portable context harness for session-based agents working on long-horizon, file-based projects.

It helps agents preserve project continuity across sessions by maintaining a small, durable set of context files instead of relying on ephemeral chat memory alone.

## Why ContextWeave Exists

Many agent workflows fail for predictable reasons:

- important project framing is forgotten
- current state and historical notes get mixed together
- milestone history becomes noisy or incomplete
- different sessions or tools drift toward different understandings of the same project

ContextWeave addresses those problems with a lightweight file model that separates:

- stable project framing
- current working state
- milestone evidence

## Core Model

ContextWeave uses three primary memory layers:

- `.contextweave/context_brief.md`
  Stable context-brief file. This changes rarely.

- `.contextweave/rolling_summary.md`
  Overwrite-style current-state snapshot. This is the main continuity file.

- `.contextweave/daily_logs/YYYY-MM-DD.md`
  Append-only milestone evidence. This is not a running scratch log.

- `.contextweave/update_protocol.md`
  Project-local override layer. This file is generated by default, but it is recommended rather than strictly required for a valid protocol `1.0` workspace.

- `.contextweave/state.json`
  Machine-readable sidecar state for revision-aware helpers and concurrency-aware checks.

The default storage mode is a hidden sidecar directory:

```text
PROJECT_ROOT/
  .contextweave/
    config.json
    state.json
    context_brief.md
    rolling_summary.md
    daily_logs/
    update_protocol.md
```

This model is intentionally file-native, portable, and easy to inspect.

ContextWeave also uses a machine-readable marker layer inside managed files.

That marker layer is the stable integration contract.
Visible markdown headings remain localizable.

For protocol `1.0`, supported workspace languages are limited to `en` and `zh-CN`.

An optional visible sidecar mode may also be used:

```text
PROJECT_ROOT/
  contextweave/
    config.json
    state.json
    context_brief.md
    rolling_summary.md
    daily_logs/
    update_protocol.md
```

Exactly one valid sidecar may exist for a project at a time.

If both `.contextweave/` and `contextweave/` exist with valid `config.json` files, the workspace is in a conflict state and should be fixed before continuing.

## What It Is Good For

ContextWeave is designed for projects that are:

- long-horizon
- file-based
- judgment-heavy
- vulnerable to cross-session drift

It currently prioritizes three profiles:

- research writing
- product document collaboration
- software project coordination

## What It Is Not

ContextWeave is not:

- a vector-memory backend
- a model-specific prompt pack
- a 24/7 autonomous execution runtime
- a full multi-agent orchestration platform
- a heavy software-engineering control plane

## Design Principles

ContextWeave is built around a few core principles:

- lightweight by default
  Use progressive disclosure instead of loading everything at once.

- file-native state
  Durable state should live in project files, not only in transient conversations.

- state/evidence separation
  Current state should not be mixed with historical evidence.

- profile-aware guidance
  Different project shapes need different emphasis, without fragmenting the core protocol.

- deterministic help where useful
  Scripts should handle setup and validation work that does not require editorial judgment.

- low project intrusion
  The default layout should stay easy to remove and avoid polluting the visible project root.

## Concurrency Boundary

ContextWeave currently supports multiple sessions over time.

For packaged helper-script writes, the current `0.1.0` release line provides a minimal hard-guard layer:

- project-scoped write locking
- atomic replace for managed file writes
- revision-aware helper commits for overwrite-style context files
- revision-aware helper appends for daily-log milestone entries

For the current `0.1.0` release line:

- read-only checks are safe to run anytime
- mutating helper scripts are single-writer operations per project
- use `preflight_context_check.py` before preparing a write
- use `commit_context_file.py` or `append_daily_log_entry.py` for revision-checked writes
- if you bypass those helpers and edit files directly, you bypass the hard concurrency guards too

## Package Layout

This is the canonical manifest for the installable `contextweave/` package.

```text
SKILL.md
README.md
USAGE.md
package-metadata.json
references/
profiles/
scripts/
LICENSE
NOTICE
```

This directory is the installable skill package.

Repository-level materials such as research notes, tests, and adapter docs are intentionally kept outside the installed skill folder.

Install the whole `contextweave/` directory into a compatible user-level or project-level skills folder.
Do not split the package apart or copy only `SKILL.md`.

This installable package is host-agnostic:

- keep the package as a single `contextweave/` folder
- place that folder wherever your host agent expects installed skills
- let the host load `SKILL.md` through its normal skill discovery flow

ContextWeave does not ship a custom launcher or host-specific adapter inside the installable package itself.

Typical installed shape:

```text
<skills-dir>/
  contextweave/
    SKILL.md
    README.md
    USAGE.md
    package-metadata.json
    references/
    profiles/
    scripts/
    LICENSE
    NOTICE
```

## Start Here

1. Read [SKILL.md](./SKILL.md).
2. Read the matching file in `profiles/`.
3. Review the shared protocol docs in `references/`.
4. Read [USAGE.md](./USAGE.md) for explicit invocation guidance.
5. Initialize a workspace sidecar with a Python `3.10+` interpreter, for example:
   - macOS/Linux: `python3.13 /path/to/contextweave/scripts/init_context.py ...`
   - Windows: `py -3.13 \\path\\to\\contextweave\\scripts\\init_context.py ...`
   The `--date` flag, when used, must be a valid ISO date in `YYYY-MM-DD` form.
   In hidden mode, initialization may also add a managed ContextWeave block to `.git/info/exclude`.
6. Use the helper scripts only for deterministic support work.
   Archive operations are preview-first and require explicit confirmation to apply.
7. Remove a workspace sidecar safely with the same Python `3.10+` interpreter when it is no longer needed.
   If unknown assets are found inside the storage root, removal now refuses by default unless you explicitly pass `--force`.
   In hidden mode, removal may also remove the managed ContextWeave block from `.git/info/exclude`.
8. Add or remove thin bridges in supported root entry files with the same Python `3.10+` interpreter.
9. See [USAGE.md](./USAGE.md) for the complete helper script map, including read-only vs write behavior and safety expectations.

For installation and invocation details, including copy/symlink examples and host expectations, see [USAGE.md](./USAGE.md).

## First Successful Run

After the package is installed, the shortest end-to-end acceptance check is:

```bash
cd /path/to/installed/contextweave
python3.13 scripts/init_context.py /absolute/path/to/project
python3.13 scripts/validate_context.py /absolute/path/to/project --json
```

If you use a different compatible interpreter, replace `python3.13` with any Python `3.10+` interpreter available in your environment.

Expected result:

- `init_context.py` creates the sidecar files
- `validate_context.py` returns JSON with `"valid": true`

Optional:

- add `--create-daily-log` if you want initialization to also create an empty daily-log scaffold for the current day
- that scaffold is only a starting structure; it should not be mistaken for proof that a real milestone already happened
- when a scaffold exists, the first real daily-log append still becomes `entry-1`
