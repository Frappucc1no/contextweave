## Project-Local Overrides

If `STORAGE_ROOT/update_protocol.md` exists, read it before applying the default playbooks below.

Project-local overrides may:

- narrow the default read order
- strengthen write rules
- tighten archive behavior
- add stricter local constraints

They do not replace the core machine-readable contract.

For protocol `1.0`, this file is a human-reviewed override layer.

It is generated by default, but it remains a recommended override file rather than a strict workspace-validity requirement.

That means:

- preflight, archive guidance, and bridge guidance should surface it clearly
- helper scripts should not silently ignore that it exists
- helper scripts do not automatically parse or execute its natural-language rules

## Cold Start

Use this flow whenever resuming a project with an existing RecallLoom structure.

1. Locate the project root.
2. Read `STORAGE_ROOT/config.json`.
3. Read `STORAGE_ROOT/state.json`.
4. Read `STORAGE_ROOT/rolling_summary.md`.
5. If `STORAGE_ROOT/update_protocol.md` exists, surface it for review before expanding beyond the minimum continuity set.
6. Read `STORAGE_ROOT/context_brief.md` only when the current task needs framing, scope, source-of-truth, or phase context that the summary does not already cover.
7. Read the latest active daily log only when milestone evidence, workday judgment, or external-writer reconciliation requires it.
8. Check whether newer artifacts exist than the maintained context files when freshness is uncertain or before a major write.
9. Prefer verified newer evidence over stale continuity files.

Cold start is for orientation, not for automatic rewriting.
Cold start should restore and recommend first.
It should not automatically continue `next_step` or execute unfinished project work without clear user confirmation.

## Choose the Write Set

Before writing anything, decide the smallest valid write set.

If `STORAGE_ROOT/update_protocol.md` exists, surface its project-local constraints for review before deciding.

Preflight guidance should expose two layers:

- primary write targets for deterministic current-state updates
- conditional review targets for files that may need review but are not default write targets
- revision context for safe helper writes

Typical mapping:

- `STORAGE_ROOT/rolling_summary.md` for durable current-state changes
- `STORAGE_ROOT/context_brief.md` for framing or phase changes
- daily log for milestone events

Daily logs should only become write targets when the current session actually creates a new milestone entry.

If only one file needs updating, update only that file.

When deterministic write safety matters, use this flow:

1. run `preflight_context_check.py` for the default quick path, or `preflight_context_check.py --full` when you intentionally want the heavier workspace artifact scan
2. prepare the content change outside the managed file
3. use `commit_context_file.py` for `rolling_summary.md`, `context_brief.md`, or `update_protocol.md`
4. use `append_daily_log_entry.py` for daily-log milestone entries

In the current `0.3.0` release line, those revision-aware write helpers do not independently reread `update_protocol.md` before every write.

## End-of-Day Update

Use an end-of-day update when:

- meaningful progress happened
- the user explicitly signals stop-work

Typical order:

1. rolling summary
2. `context_brief.md` if needed
3. today's daily log

Always leave a clear next-step note when possible.

## Compression

Compress `rolling_summary.md` when it becomes hard to scan.

The compression rule is:

1. keep current state
2. move supporting detail to the daily log
3. drop stale or false content
4. preserve the next step and unresolved risks

Compression should make the file easier to resume from, not more abstract.

## Archive

Archive old daily logs when:

- the log set grows too large
- a major phase boundary is crossed

Archived logs should remain retrievable on demand, but fall out of the default cold-start path.

Archive actions should be preview-first.

Recommended behavior:

- preview targets first
- require explicit confirmation before moving files
- refuse the whole operation if archive-target collisions are detected
- if `update_protocol.md` exists, surface it for review before archiving

## External-Writer Reconciliation

If another tool or session may have updated the files:

1. reread current files
2. identify new facts or judgments
3. merge before writing
4. never discard append-only milestone entries

The goal is conservative continuity, not aggressive overwriting.
